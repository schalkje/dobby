## Why

Dobby's PBI lifecycle skills (`dobby-create-pbi`, `dobby-close-pbi`, `dobby-propose-from-pbi`) are hard-wired to Azure DevOps. We want the same conversational flows ("close this", "spec from PBI 42") to work on projects that track issues in GitHub instead. The two trackers are semantically interchangeable for our purposes — a PBI and a GitHub Issue are the same kind of object — but the APIs, field shapes, and supporting concepts (areas, iterations, attachments, dev-links) differ enough that a single skill cannot serve both without a clean routing layer.

This change introduces that routing layer: a per-project setting that selects the backend, and a split of each workflow into a thin dispatcher plus a backend-specific implementation. It mirrors, on the *target-system* axis, the host-axis abstraction we already have for Claude Code and Copilot CLI.

## What Changes

- Add `.dobby/config.json` as the single source of project tracker configuration, with a `backend` selector (`"ado"` or `"github"`) and per-backend connection blocks. **BREAKING**: replaces `.dobby/azdo-defaults.json`.
- A one-time migration lifts the contents of `.dobby/azdo-defaults.json` into the `ado` block of `.dobby/config.json` and removes the old file.
- The three existing skills (`dobby-create-pbi`, `dobby-close-pbi`, `dobby-propose-from-pbi`) become **dispatchers**: they read `.dobby/config.json` and hand off to a backend-specific sibling skill. Their names are preserved so user vocabulary doesn't change.
- Six new backend skills are introduced:
  - `dobby-ado-create-pbi`, `dobby-ado-close-pbi`, `dobby-ado-propose-from-pbi` — contain today's logic, extracted from the dispatchers.
  - `dobby-gh-create-issue`, `dobby-gh-close-issue`, `dobby-gh-propose-from-issue` — net-new GitHub implementations using the `gh` CLI.
- The GitHub close-issue flow **requires an open Pull Request**. Screenshots and other evidence are committed to the PR branch under `docs/evidence/issue-<N>/` and embedded in the PR description; the PR's `Closes #<N>` directive closes the issue on merge.
- New helper script `scripts/migrate-dobby-config.py` performs the one-time `azdo-defaults.json` → `config.json` migration; idempotent and safe to re-run.
- `scripts/sync-skills.py` continues to mirror canonical sources to `.github/skills/` and `.claude/skills/`; no logic change required beyond picking up the new skill directories.

Not in scope:
- Multi-tracker projects (a single project cannot mix ADO and GitHub).
- Per-invocation backend override flags.
- Migration of historical PBIs to GitHub Issues or vice versa.

## Capabilities

### New Capabilities

- `backend-routing`: How dobby skills read the project tracker config and route conversational intents to the correct backend implementation. Covers the `.dobby/config.json` schema, dispatcher-skill behavior, missing-config prompting, and the migration from `azdo-defaults.json`.
- `github-issue-creation`: How `dobby-gh-create-issue` creates a GitHub Issue from a conversational request, including markdown body shape (description + acceptance criteria in one body), label and milestone assignment, and parent linkage via task-list references.
- `github-issue-closing`: How `dobby-gh-close-issue` closes an issue through its accompanying Pull Request. Requires an open PR, commits screenshot evidence to the PR branch, embeds evidence in the PR description, and relies on `Closes #<N>` for issue closure on merge.
- `github-spec-proposal`: How `dobby-gh-propose-from-issue` fetches a GitHub Issue and generates an OpenSpec change directory seeded from it, parallel to today's PBI-based flow.

### Modified Capabilities

- `pbi-creation`: Narrows scope to "applies when `backend == ado`". The user-facing entry point becomes the `dobby-create-pbi` dispatcher; the implementation moves to `dobby-ado-create-pbi`. No requirement-level behavior change beyond the scope clause.
- `closing-evidence`: Narrows scope to "applies when `backend == ado`". The implementation moves to `dobby-ado-close-pbi`. GitHub closure is covered by the new `github-issue-closing` capability and has a different evidence model (PR-resident files, not work-item attachments).
- `before-capture`: Narrows scope to "applies when `backend == ado`". GitHub's evidence flow uses Playwright output committed directly to the PR branch and does not maintain a separate before/after store.

## Impact

- **Skills**: 3 existing skills become dispatchers; 6 new sibling skills introduced. All under `skills/` with mirrored copies regenerated by `sync-skills.py` into `.github/skills/` and `.claude/skills/`.
- **Config files**: `.dobby/azdo-defaults.json` is removed in favor of `.dobby/config.json`. Migration script handles existing checkouts.
- **Helper scripts**: ADO Python helpers (`azdo-update-fields.py`, `azdo-add-comment.py`, `azdo-upload-attachment.py`, `azdo-add-dev-links.py`, `evidence-store.py`) remain unchanged and live under the new `dobby-ado-*` skills. New helper script for config migration. GitHub flows rely on the `gh` CLI; no new Python helpers are anticipated.
- **External dependencies**: GitHub flows require the `gh` CLI to be installed and authenticated. ADO flows continue to require `az` with the `azure-devops` extension.
- **Documentation**: `CLAUDE.md` updated to describe the dispatcher + backend-skill architecture and the new config file.
- **No code changes outside `skills/`, `scripts/`, `.dobby/`, and documentation.** This is purely an additive change at the workflow layer.
